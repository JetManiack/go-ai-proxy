# vLLM (gemma-4-31B-it) — руководство для разработчика клиента

Гайд по интеграции приложений с нашим LLM-сервером. Описывает и базовые вещи про vLLM,
и наши конкретные особенности, которые влияют на поведение и латентность приложения.

> TL;DR: OpenAI-совместимый API на `https://vllm.local/v1`, одна модель `gemma-4-31b-it`,
> контекст до **224k токенов**, скорость **~3 токена/с**, есть **поток мыслей (reasoning)** и
> **function calling**. Главное — закладывайте **большие таймауты**, используйте **стриминг**
> и **переиспользуйте системный промпт** (кэшируется).

---

## 1. Что это и как подключиться

Сервер — **vLLM**, движок инференса с **OpenAI-совместимым** REST API. Можно использовать
официальный OpenAI SDK (Python/JS/…), просто подменив `base_url`.

| Параметр | Значение |
|---|---|
| Base URL | **`https://vllm.local/v1`** (обязательно с `/v1`!) |
| Альтернатива без TLS | `http://vllm.local/v1` или `http://192.168.10.250:30800/v1` |
| API-key | не требуется (передавайте любую непустую строку, SDK часто требует поле) |
| Модель (`model`) | `gemma-4-31b-it` |
| Протокол | HTTP/1.1, JSON; стриминг — SSE (`text/event-stream`) |

**TLS.** Сертификат `vllm.local` подписан **нашим внутренним CA**. Чтобы не было ошибок
рукопожатия (`error sending request` / 502), на хосте приложения нужно **доверять корневому
сертификату** (выдаётся отдельно, `vllm-lan-root.crt`):
- Node.js: `NODE_EXTRA_CA_CERTS=/path/vllm-lan-root.crt`
- система (Debian/Ubuntu): положить в `/usr/local/share/ca-certificates/` + `update-ca-certificates`
- Python `requests/httpx`: `verify="/path/vllm-lan-root.crt"`
- быстрый обход на время разработки — использовать `http://…` (без TLS) или отключить проверку.

**Доступные эндпойнты:** `/v1/chat/completions`, `/v1/completions`, `/v1/models`.
Эмбеддингов нет (модель генеративная). Health: `/health`. Метрики: `/metrics`.

Быстрый старт:
```bash
curl https://vllm.local/v1/chat/completions -H 'content-type: application/json' -d '{
  "model":"gemma-4-31b-it",
  "messages":[{"role":"user","content":"Привет!"}],
  "stream":true
}'
```
```python
from openai import OpenAI
client = OpenAI(base_url="https://vllm.local/v1", api_key="x")  # ключ не проверяется
for chunk in client.chat.completions.create(
        model="gemma-4-31b-it",
        messages=[{"role":"user","content":"Привет!"}],
        stream=True):
    print(chunk.choices[0].delta.content or "", end="")
```

## 2. Лимиты по токенам

- **Контекстное окно: 224k токенов** (`max_model_len = 229376`). Это **общий лимит на
  `prompt + completion`**, не на каждое по отдельности. Токен ≈ 3–4 символа.
- `max_tokens` в запросе ограничивает **только генерацию**. Если `prompt + max_tokens > 224k` —
  получите **HTTP 400** (`BadRequestError`, `maximum context length…`). Уменьшайте промпт или
  `max_tokens`.
- Если не указать `max_tokens`, модель пишет до лимита контекста или до стоп-условия. Для
  reasoning-запросов **обязательно ставьте `max_tokens`** с запасом (мысли едят токены, см. §5).
- Считать токены заранее: токенизатор Gemma (через `transformers`/`tokenizers`), либо опираться
  на `usage` в ответе (`prompt_tokens` / `completion_tokens` / `total_tokens`).
- Архитектурный максимум модели — 256K, но у нас стоит 224k (упирается в VRAM под KV-кэш).

## 3. Производительность, параллелизм, таймауты ⚠️ важно для UX

Сервер работает на одном встроенном GPU (AMD Strix Halo), память **bandwidth-bound**. Это
определяет всё поведение:

- **Скорость декода ~3 токена/с.** Ответ на 300 токенов ≈ 100 секунд. Это не баг — таково
  железо. (В планах int4-квантизация → ~×4 быстрее.)
- **Параллелизм.** Жёсткого лимита запросов нет, но они делят общий KV-кэш (~280k токенов) и,
  главное, **общую пропускную способность памяти**. Суммарная скорость почти постоянна:
  2 запроса параллельно → каждый ~1.5 т/с, и т.д. По памяти влезут десятки коротких запросов,
  но для нормальной латентности на пользователя реально **1–4 активных**. Не «долбите» сервер
  множеством параллельных запросов — это не ускорит, а замедлит каждый.
- **Prefill больших промптов медленный.** Обработка входных токенов до первого выходного (TTFT)
  на длинном контексте — это **десятки секунд–минуты** при 50k+ входных токенов.
- **Таймауты.** Ставьте клиентские таймауты **щедро — минуты, не секунды** (особенно для не-стрим
  запросов и больших контекстов). Дефолтные 30–60с почти гарантированно оборвут длинную
  генерацию.
- **Стриминг — почти обязателен** для интерактивного UX: пользователь видит токены по мере
  генерации, а не ждёт 2 минуты «вникуда». `"stream": true`; usage в конце — через
  `"stream_options": {"include_usage": true}`.
- **Кэш префикса включён (prefix caching).** Если много запросов делят **одинаковое начало**
  (системный промпт, общий контекст) — повторный prefill этой части почти бесплатен. Практика:
  **держите системный промпт стабильным и в начале** — это заметно ускоряет TTFT.
- **Очередь.** Запросы сверх ёмкости встают в очередь (не отбрасываются) — но ждут. Следите за
  `vllm:num_requests_waiting` в метриках, если нужен мониторинг нагрузки.

## 4. Sliding window attention (про длинный контекст)

Gemma использует **гибридное внимание**: большинство слоёв — **локальное «скользящее окно»
(sliding window) 1024 токена**, и лишь часть слоёв — **глобальное внимание** на весь контекст.

Что это значит для приложения:
- **KV-кэш дешёвый** → поэтому и влезает контекст 224k на скромной памяти. Это плюс.
- Большинство слоёв «видят» локально только ~1024 последних токена; дальнюю связь держат редкие
  глобальные слои. На практике модель работает с длинным контекстом, но **не рассчитывайте, что
  каждая мелкая деталь из начала 200k-промпта будет одинаково «свежа»**, как в коротком запросе.
- Вывод: **кладите важное ближе к концу промпта / в системную инструкцию**, не растворяйте
  ключевые требования в гигантском контексте. Это общая хорошая практика, но здесь особенно.

## 5. Поток мыслей (Reasoning / Chain-of-Thought)

Модель умеет «думать» отдельным каналом перед ответом. Это даёт качество на сложных задачах,
но **стоит токенов и времени**.

**Как включить (на запрос):**
```json
{
  "model":"gemma-4-31b-it",
  "messages":[{"role":"user","content":"..."}],
  "chat_template_kwargs": {"enable_thinking": true},
  "stream": true,
  "max_tokens": 1024
}
```

**Где «мысли» в ответе — поле `reasoning` (НЕ `reasoning_content`!):**
- не-стрим: `choices[0].message.reasoning` (мысли) и `choices[0].message.content` (ответ);
- стрим: сначала идут чанки `choices[0].delta.reasoning`, потом `choices[0].delta.content`.

⚠️ Это **нестандартное** имя поля. Многие UI/SDK ждут `reasoning_content` (как у OpenAI o1 /
DeepSeek) и **не покажут мысли**. Если используете готовый UI — настройте маппинг `reasoning`,
или отображайте его сами.

**Авто-включение.** В шаблоне модели thinking включается сам, если в запросе есть **`tools`** или
**первое сообщение — `system`/`developer`**. То есть агентные запросы (с инструментами/системным
промптом) **думают всегда**, даже без `enable_thinking`. Передача `enable_thinking:false`
**не отключит** размышления, если присланы `tools`.

**Практика:**
- Хотите быстрее/дешевле и задача простая — **не** включайте thinking (и без tools/system).
- Включили reasoning — **обязательно `max_tokens` с запасом** (мысли + ответ), иначе ответ
  оборвётся на середине размышления (`finish_reason: "length"`, `content` пустой).
- Для UX показывайте мысли в сворачиваемом блоке «Размышление…», ответ — отдельно.

## 6. Function calling (инструменты)

Поддерживается OpenAI-совместимый tool calling.

```json
{
  "model":"gemma-4-31b-it",
  "messages":[{"role":"user","content":"Погода в Париже?"}],
  "tools":[{"type":"function","function":{
    "name":"get_weather",
    "description":"Current weather for a city",
    "parameters":{"type":"object","properties":{"location":{"type":"string"}},"required":["location"]}
  }}],
  "tool_choice":"auto"
}
```
- Ответ при вызове: `finish_reason:"tool_calls"`, аргументы в
  `choices[0].message.tool_calls[].function.arguments` (JSON-строка).
- Дальше — стандартный цикл: выполняете функцию, добавляете сообщение `role:"tool"` с
  результатом, вызываете снова.
- **Reasoning + tools:** при наличии `tools` модель сначала «думает» (поле `reasoning`), затем
  выдаёт `tool_calls`. Закладывайте это в обработку и в `max_tokens`.

## 7. Мультимодальность (картинки)

Модель принимает изображения (OpenAI vision-формат):
```json
{"role":"user","content":[
  {"type":"text","text":"Что на картинке?"},
  {"type":"image_url","image_url":{"url":"data:image/jpeg;base64,...."}}
]}
```
Учтите: обработка изображения добавляет входные токены и время prefill.

## 8. Чек-лист «граблей»

- [ ] Base URL **с `/v1`** (`https://vllm.local/v1`). Без него — 404 на `/models`.
- [ ] **Доверие к нашему CA** на хосте приложения (или `http://`). Иначе 502 / TLS-ошибка.
- [ ] **Таймауты — минуты**, не секунды. Декод ~3 т/с.
- [ ] **Стриминг** для интерактива; иначе долгое ожидание без обратной связи.
- [ ] Reasoning — поле **`reasoning`**, не `reasoning_content`. И ставьте `max_tokens`.
- [ ] Не распараллеливайте агрессивно — латентность на запрос только вырастет.
- [ ] Стабильный системный префикс → быстрее за счёт prefix-кэша.
- [ ] `prompt + max_tokens ≤ 224k`, иначе 400.
- [ ] Модель одна (`gemma-4-31b-it`); эмбеддингов нет.
- [ ] Пустой/«перемешанный» ответ — это баг сборки на клиенте, не сервер (см. §10).

## 9. Справка по полям ответа

```jsonc
{
  "choices":[{
    "message":{
      "content": "...",          // ответ (null, пока модель «думает» или при tool_calls)
      "reasoning": "...",         // ⚠️ поток мыслей (наша особенность; не reasoning_content)
      "tool_calls":[ ... ]        // при function calling
    },
    "finish_reason":"stop"        // stop | length | tool_calls
  }],
  "usage":{"prompt_tokens":N,"completion_tokens":M,"total_tokens":N+M}
}
```
В стриме то же приходит инкрементально в `choices[0].delta.{content,reasoning,tool_calls}`.

## 10. Отладка: пустой или «перемешанный» ответ ⚠️ почти всегда баг клиента

Если пользователь видит **пустой** ответ или **перемешанный** текст (символы/куски слов не в
том порядке) — сервер тут, как правило, ни при чём. vLLM отдаёт токены **строго по порядку** и
возвращает `200 OK`; проблема почти всегда в том, как клиент собирает и отображает ответ.

**Пустой ответ.** Чаще всего ответ не в `content`, а в другом поле:
- модель «думает» → текст в **`reasoning`** (см. §5), а `content` пуст;
- модель вызвала инструмент → ответ в **`tool_calls`**, `finish_reason:"tool_calls"`, `content`
  пуст (см. §6);
- `max_tokens` мал и мысли съели весь бюджет → `finish_reason:"length"`, `content` пуст (§5).

  Чините так: рендерьте **`content || reasoning`**, обрабатывайте `tool_calls`, проверяйте
  `finish_reason`, и для reasoning-запросов давайте `max_tokens` с запасом. UI, показывающий
  **только `message.content`**, будет периодически показывать пустоту — это ожидаемо.

**Перемешанный текст** (напр. «профессиональны*йчик разработ*» вместо «…ный разработчик»).
Это **рассинхрон сборки SSE-стрима на стороне клиента** — чанки склеиваются не в порядке прихода.
vLLM такого не делает: сырой стрим приходит строго последовательно
(`'про' 'фессиона' 'льный' ' разработ' 'чик' …`). Типичные причины и фиксы:
- конкурентная/асинхронная обработка чанков → **накапливайте `delta.content` строго в порядке
  получения**, в одном буфере, без параллелизма;
- смешивание каналов → держите **`reasoning`, `content` и `tool_calls` в раздельных буферах**,
  не сливайте поток мыслей в основной текст.

**Сверяете с логами vLLM?** Сервер пишет время в **UTC**. Если ваш UI показывает локальное время
(напр. MSK = UTC+3), запрос будет в логах на 3 часа «раньше» — это не «сервер промолчал», а
смещение часового пояса. Сопоставляйте по UTC.
